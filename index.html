iOS Unit自动化测试框架Quick简单使用

Posted on 2018-02-05 | Edited on 2018-10-14

前言

iOS自动化测试一直不如Android的方便，最近没事看简单看了下，分别基本使用了一下自动化框架Quick和KIF，做一些简单的介绍。

安装

  • 安装前设置
    在安装之前将工程中的Defines Module设为YES，如下图：
  • 使用cocoapods安装Quick和Nimble。在Podfile中添加pod命令，如下通过def xxx end来包裹pod命令，这样该def可以被用于多个相同的target，如UnitTest和UITest

Quick在Xcode9上Swift4貌似支持的不好，需将Quick和Nimble的版本手动配置为Swift3.2，后续官方可能会优化对Swift4的支持

pod install安装即可使用
注：可以安装Quick模板，这样可以快速的新建Quick测试文件，具体步骤查看GitHub官方文档，但是安装好了总觉得Xcode变得很慢，有可能是需要安装包管理的原因，对Xcode9支持的也不好，所以其实最好不安装模板，就几行代码，可以Copy到一个地方，创建的时候Paste过来即可。

出现No Such Module ‘Quick’问题解决

这个问题也得归结于目前对Xcode9 Swift4支持的不好，可以根据GitHub的issue #729（closed里面）进行尝试解决，不然编译通不过，对于入门者没有自动提示也是呵呵哒。issue #729主要 解决方法总结为：

  1. 手动将Quick版本配置到Swift3.2，上述Podfile中已提及
  2. 注意使用最新的Quick版本，即默认的master分支最新源码
  3. Pod install后再次检查，将工程中的Defines Module和pod工程中的Defines Module设置为YES
  4. framework search paths不配置任何东西

还有可以尝试按照GitHub官方Document里面的TroublesShooting(常见安装问题)进行解决，其总结为：

  1. 清掉Xcode DriveData数据
  2. 通过Manage Scheme将Quick、Nimble、Pods-ProjectNameTests勾选上，让他们show出来，然后分别选中他们各编译一次，最后返回你的工程进行测试

我的做了以上尝试当天其实还是一直显示No Such Module ‘Quick’，然后就没管它去看了KIF UI 测试框架，结果第二天来就可以找到了，确实这个我是没啥好说的，肯定的是Quick对Xcode9/Swift4的支持不够好，还有可能是Xcode本身的indexing系统在某些时候不理想（Xcode9当然是目前indexing最好的版本，其本身用Swift重构后解决了以前的太多不方便）

Test文件中不能引用主工程类问题解决

Test文件中找不到主工程中的类，如测试其ViewController，需要将ViewController作为实例变量。尝试了几种方法，如import ProjectName，将Target设置到Test
解决方法，参照官方的XCTest的写法，这样引入@testable import ProjectName即可

写在结尾

对iOS自动化测试研究不深，平时也很少用到，希望能够有更好的测试框架和更简易的教程出来。

DLNA CyberLink使用

Posted on 2016-03-25 | Edited on 2018-10-13

前言

参与公司的一个项目，手机作为遥控器，与公司的硬件产品（智能机顶盒，类似小米盒子、天猫魔盒等）进行配合，通过手机控制电视上的操作，如前后左右、关机、音量大小、返回、主页、菜单等。硬件采用的协议是DLNA协议，是基于PlatinumKit（C++）写的。由于该项目的时间比较短，顾没有去使用PlatinumKit，因为那需要去进一步研究相关的C++代码，需要一定的时间。在Github上搜索了一下，现成的IOS三方库就只有FuruyamaTakeshi/DLNA和wangshuaidavid/DLNA_iOS_Platinum，而后者只实现了DMS，没有实现DMC，对于手机遥控器来说需要的正是DMC，所以采用了前者。

导入整个clinkc工程使用

DLNA CyberLink是基于C语言写的一套代码，存在很多的依赖（头文件），它只提供了一个clinkc.xcodeproj的工程文件，并没有向其他三方库一样有单独出来的一套容易使用的文件目录，而且也很难去单独剥离出来（是可以的，我也尝试了），最后选择了将整个工程导入，只使用其中可以用到的IOS封装。
需要注意一下两点：

  1. 将导入的clinkc.xcodeproj的Build Settings下的Architectures修改为ARCHS_STANDARD也就是支持armv7和arm64的（保持和你的原有项目一直，现在的机子都支持arm64，选ARCHS_STANDARD就对了），在相应的Valid Architectures下也要和主项目一致，如图:

  2. 两个Build Settings下Development中的Skip Install要设置正确，clinkc.xcodeproj中设置为YES，在自己的主工程中设置为NO，这也就标示了你的工程是这工程，如图:

写在结尾

大致的使用就是上面这些了，使用起来还是挺简单的。但是它也真的坑了我很久，一开始是导入整个工程的时候死活包架构错误，然后它是基于同步的，在一定程度上需要依靠良好的网络，会有一定的卡顿现象，我甚至去改过它的socket连接异步…就现成的三方库来说，它没有选择，但是对于一个产品来说，这样的效果可能还是会影响用户的体验，有时间和需求的话，我会采用PlatinumKit进行重新封装，达到更好的效果。
